PropGenius AI — CI/CD Setup Guide

Next.js 16 GitHub Actions Vercel

This document covers the complete CI/CD pipeline setup for the PropGenius AI project, including GitHub Actions configuration and Vercel deployment.

1. Architecture Overview

The CI/CD pipeline has two layers that work independently:

LayerPlatformPurposeTrigger
CI (Code Quality)GitHub ActionsLint, type-check, build verificationPush / PR to main
CD (Deployment)VercelBuild & deploy to productionPush to main
How they interact: GitHub Actions validates your code quality. Vercel handles the actual deployment. Both run in parallel — they don't depend on each other.

2. GitHub Actions Pipeline

2.1 Pipeline File

Location: .github/workflows/ci.yml

name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npm run lint
      - run: npm run type-check
      - run: npm run build
        env:
          NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.NEXT_PUBLIC_SUPABASE_URL }}
          NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.NEXT_PUBLIC_SUPABASE_ANON_KEY }}
          NEXT_PUBLIC_APP_URL: ${{ secrets.NEXT_PUBLIC_APP_URL }}

2.2 What Each Step Does

StepCommandPurpose
Checkoutactions/checkout@v4Clones your repo code
Setup Nodeactions/setup-node@v4Installs Node.js 20 with npm cache
Installnpm ciClean install of dependencies from lockfile
Lintnpm run lintChecks code style & catches errors (ESLint)
Type Checknpm run type-checkValidates TypeScript types (tsc --noEmit)
Buildnpm run buildFull Next.js production build

3. Setting Up GitHub Secrets

Important: Use Secrets (not Variables) because these contain sensitive API keys. Secrets are encrypted and never shown in logs.

1 Go to your GitHub repository

2 Navigate to Settings → Secrets and variables → Actions

3 Click "New repository secret" and add each secret:

Secret NameValue (from your .env.local)
NEXT_PUBLIC_SUPABASE_URLYour Supabase project URL (e.g., https://xxx.supabase.co)
NEXT_PUBLIC_SUPABASE_ANON_KEYYour Supabase anon/public key
NEXT_PUBLIC_APP_URLYour app URL (e.g., https://propgenius.vercel.app)
Why only NEXT_PUBLIC_* vars? These are inlined at build time by Next.js. Server-only secrets (like SUPABASE_SERVICE_ROLE_KEY, TWILIO_AUTH_TOKEN) are NOT needed for CI — they're only used at runtime.

4. Vercel Deployment Setup

1 Go to vercel.com and import your GitHub repository

2 Vercel auto-detects Next.js — no build config needed

3 Add ALL environment variables in Vercel dashboard (Settings → Environment Variables):

NEXT_PUBLIC_SUPABASE_ANON_KEYSupabase client (browser)

5. How the Full Flow Works

Developer pushes code to main
        │
        ├──→ GitHub Actions (runs in ~1-2 min)
        │      ├── npm ci
        │      ├── npm run lint        → catches code errors
        │      ├── npm run type-check  → catches type errors
        │      └── npm run build       → catches build errors
        │
        └──→ Vercel (runs in ~1-2 min)
               ├── Installs dependencies
               ├── Builds the app
               └── Deploys to production URL
Both pipelines run in parallel. If GitHub Actions fails, Vercel still deploys (and vice versa). To block deploys on CI failure, enable "Required status checks" in GitHub branch protection rules.

6. Optional: Branch Protection

To prevent broken code from reaching production, enable branch protection:

  1. Go to GitHub → Settings → Branches
  2. Click "Add branch protection rule"
  3. Branch name pattern: main
  4. Enable: "Require status checks to pass before merging"
  5. Select the "build" check (from GitHub Actions)
  6. Save changes

This ensures no PR can be merged to main unless lint, type-check, and build all pass.

7. Troubleshooting

CI build fails with "missing environment variable"

Make sure all 3 NEXT_PUBLIC_* secrets are added in GitHub (Settings → Secrets → Actions). Names must match exactly.

Vercel deploy fails

Check Vercel dashboard → Deployments → click the failed deploy → read the build logs. Most common cause: missing environment variables in Vercel settings.

CI passes but Vercel fails (or vice versa)

They use different environments. Ensure environment variables are set in both GitHub Secrets and Vercel dashboard.

Pipeline doesn't trigger

The CI only triggers on pushes and PRs to the main branch. If you're pushing to a different branch, it won't run.
